SRTE

Introduction

Welcome to the Sharable Content Object Reference Model (SCORM) 2004 4th Ed. SRTE Version 1.1.1. The SRTE provides an example implementation of the concepts described in SCORM. Specifically, it provides an example of the SCORM Run-Time Environment (Application Programming Interface (API) and Data Model) and SCORM Sequencing and Navigation as they might be applied in a learning management system (LMS) environment. Robust exception and error handling are not included in this sample code, nor were performance optimizations considered.

Even though there are a number of valid web-enabled SCORM implementations, this version of the SRTE was implemented as a web-based client/server application using HTML, JavaScript, JavaServer Pages (JSP), Java Applets and Java Servlets. Refer to the "Design Information" section of this Readme for additional SRTE design information.

Installation


Prerequisites

IMPORTANT: Read ALL of the Installation Instructions BEFORE beginning installation.

This release of the SRTE requires the Java Development Kit (JDK) from any one of the following Java versions:

  • Java SE 6, Update 1 or later

The latest version of this software can be downloaded free of charge at http://java.sun.com/javase/downloads/index.jsp. Archived versions are available at http://java.sun.com/products/archive/. Refer to this site for a list of specific Java versions verified.

Notes:

  • The SRTE requires the full Java Development Kit. The Java Runtime Environment (JRE) alone is NOT sufficient for this software.
  • The SRTE client may be accessed from a machine other than the one on which it is installed using Microsoft® Internet Explorer®. In this case, the Java Runtime Environment must be installed on the second machine.
After installing the Java Development Kit, you must:
  1. Create an environment variable named "JAVA_HOME" and set its value to the root directory of the Java Development Kit.
    • For Windows XP: Right click "My Computer", select "Properties", select the "Advanced" tab and then click "Environment Variables". Add a JAVA_HOME entry in the User or System Variables that points to the location where you installed the Java Development Kit.
    • For Windows Vista: Open the "Start" menu and right click "Computer", select "Properties", select the "Advanced System Settings" link on the left navigation menu, and then click Environment Variables. Add a JAVA_HOME entry in the User or System Variables that points to the location where you installed the Java Development Kit.

  2. Add the full path of the Java Development Kit bin directory (where the java.exe is located) to the PATH environment variable.
    • For Windows XP: Right click "My Computer", select "Properties", select the "Advanced" tab and then click "Environment Variables". Add the Java Development Kit bin directory to the Path by adding a semicolon followed by a reference to the JAVA_HOME environment variable followed by "\bin" (i.e., %JAVA_HOME%\bin).
    • For Windows Vista: Open the "Start" menu and right click "Computer", select "Properties", select the "Advanced System Settings" link on the left navigation menu, and then click Environment Variables. Add the Java Development Kit bin directory to the Path by adding a semicolon followed by a reference to the JAVA_HOME environment variable followed by "\bin" (i.e., %JAVA_HOME%\bin).

Installation

IMPORTANT: Only install the SRTE after installing the Java Development Kit as described in the Prerequisites.

The SRTE is available for download as a self-extracting executable install file. Once you have completed downloading the file, perform the following steps:

  1. Double-click the downloaded self-extracting executable file to begin the installation process.
  2. Click Next to continue from the "Welcome" page.
  3. Read the "License Agreement"
    • Click Yes to agree and continue with the install.
    • Click No if you do not agree with the terms of the license and wish to exit the install.
  4. Choose whether you would like the SRTE to be installed for Anyone who uses this computer (all users) or Only me.
  5. The recommended root installation directory is "C:\ADL". You may choose another root installation directory, but the SRTE install will always append the "\ADL\SCORM_2004_4th_Ed_Sample_RTE_V1.1.1" sub-directory to the specified root installation directory. The remainder of this Readme document assumes that the SRTE will be installed to "C:\ADL\SCORM_2004_4th_Ed_Sample_RTE_V1.1.1". This is referred to as "<drive>:\<install_directory>". If you override the default to use a different root installation directory, please remember the directory you selected when reading the remainder of this Readme.
  6. Click Next
  7. Click Install to begin the installation
  8. You will have the option to view a copy of this Readme document.
    • Check "Show the Readme file" to view the Readme.
    • Uncheck "Show the Readme file" to skip viewing the Readme.
  9. Click Finish to complete the installation.
Note: This installation will not remove or un-install previous versions of the SRTE.

Note: The apache-tomcat-5.5.20 servlet engine and web server will be installed and pre-configured to bind to port 8080.

Revisions
Changes from SCORM 2004 4th Ed. Sample RTE Version 1.1 to SCORM 2004 4th Ed. Sample RTE Version 1.1.1
  • The SRTE course import process was updated to enforce the following rules:
    • Parents of a given element are correct in both name and XML namespace.
  • The SRTE was also updated to:
    • Correctly display user status information when no courses exist.
    • Provide the user with the options to reset or sort courses.
Changes from SCORM 2004 4th Ed. Sample RTE Version 1.0 to SCORM 2004 4th Ed. Sample RTE Version 1.1
  • The SRTE course import process was updated to enforce the following rules:
    • Children of the <imsss:sequencing> element must exist only under an <imsss:sequencing> element.
  • The SRTE Sequencer was updated to:
    • Determine if the root was suspended before deleting the Data Store objects.
    • Check the state of each activity before performing post condition evaluations.
    • Correctly process "adl.nav.request_valid" GetValue() and SetValue() requests.
    • Store and return "incomplete" when "not attempted" is set on secondary objectives and later retrieved via a shared global objective.
  • The SRTE was also updated to:
    • Handle all course management activities from one central Home Page. Course import, registration, launching, and status viewing were combined into one central location which also contains links to administrative options and user management options.
Changes from SCORM 2004 3rd Edition Sample RTE Version 1.0.2 to SCORM 2004 4th Ed. Sample RTE Version 1.0
  • The SRTE course import process was updated to enforce the following rules:
    • If the "xsi:schemaLocation" attribute is used, then every namespace used in the XML instance must be listed in that attribute.
    • If the "xsi:schemaLocation" attribute is used, the actual locations of the attribute must be checked, even if they are not in the root.
    • The <adlnav:presentation> element must have a parent that has an "identifierref" attribute that refers to a resource with "scormtype" attribute equal to "sco" or "asset".
    • The <adlcp:timeLimitAction>, <adlcp:completionThreshold>, and <adlcp:dataFromLMS> elements shall only be used with <item> elements that reference resources with the "scormtype" attribute equal to "sco".
    • All extension elements must be valid against the controlling documents(XSDs) provided.
    • The correct multiplicity on the <ruleCondition> element beneath any of the <preConditionRule>,<postConditionRule>, <exitConditionRule> elements must be one and only one.
    • An "ID" attribute cannot be allowed on a <sequencing> element that is a child of an <item> or <organization> element.
    • The <targetObjectiveID> element must not contain only white space characters.
    • Packages that contain (sub)manifests in the <manifest> element (or its children) will fail to import when Validation is enabled. In cases that a (sub)manifest exists, but isn't referenced from the <manifest> element, it will be ignored during execution.
    • Packages containing <resource> "href" attributes that reference external URLs will fail to import.
    • Packages that contain sequencing information (The use of "flow = false" and "choice = false" Control Modes on the package root with no reachable children) which will result in no reachable activities at the time of course launch will fail to import.
    • Objective IDs which differ only in case must be treated as unique objective IDs. (ex: "OBJ1" is a different ID than "obj1")
  • The SRTE Sequencer was updated to:
    • If an <item> contains a <imsss:sequencing> element that has an IDRef to a <imsss:sequencing> element in an <imsss:sequencingCollection> then the elements in the sequencingCollection are now properly merged into the sequencing information for the item.
    • Behave correctly regarding the use of the "constrainChoice = true" attribute on the <adlseq:constrainedChoiceConsiderations> element on the root of the course.
    • Correctly initializes a cluster activity and process the default Rollup Rules.
    • Correctly handle Activity status "unknown" so that this invalid state does not cause a Not "Satisfied" or Not "Completed" rule from triggering in the LMS.
  • The SRTE was also updated to:
    • Properly process Extended Rollup based on status changes to the secondary objectives.
    • Allow the "Quit" button to properly persist " completion status" and "success status" data.
    • Directly launch the single leaf Activity in content packages that contain only a single SCO.
    • Disable all Navigation Controls while the course content is being loaded or unloaded.
    • Correctly process the whitespace component of identifier, indentifierref, ID, and IDRef attributes based on the defined datatypes of those attributes.
 
Running the SRTE

Instructions

These instructions are for running the SRTE on a Windows platform. Windows is the only platform on which the software has been tested and for which instructions are provided.

To Run the SRTE:

Ensure that your default browser is Microsoft Internet Explorer 6.0 or greater and then start the apache-tomcat-5.5.20 servlet container and Web server that is provided with this installation by:

  1. Clicking the shortcut at Start Menu -> All Programs -> ADL -> SCORM_2004_4th_Edition_Sample_Run-Time_Environment_version_1.1.1 -> Start Server. A command prompt window will appear and show several lines of textual information. This window will remain open until you stop the server.
  2. Clicking the Start Menu -> All Programs -> ADL -> SCORM_2004_4th_Edition_Sample_Run-Time_Environment_version_1.1.1 -> SCORM 2004 4th Edition Sample Run-Time Environment version 1.1.1 will launch the SRTE Main Page.
  3. Log in using
    • username: "admin"
    • password: "admin"

Online help pages are available once the SRTE is started. "Help" links will typically be located on the top right of each page.

To close the SRTE, first close the browser in which it is running. To stop the apache-tomcat-5.5.20 servlet container and Web server, click Start Menu -> All Programs -> ADL -> SCORM_2004_4th_Edition_Sample_Run-Time_Environment_version_1.1.1 -> Stop Server.

To return the SRTE to a "clean" state, log into the SRTE and from the home page choose "Clear Database". This will remove all of the imported course files as well as any information associated with them. It will also remove all user registration information and all Global Objective information.

Course Navigation User Interface

In addition to any User Interface (UI) controls provided within course content, the SRTE provides several mechanisms for navigating through a course that complies with Section 2.1.7 of the SCORM 2004 4th Ed. Testing Requirements. These mechanisms include the navigation buttons in the top frame of the SRTE, such as the Quit, Suspend, Previous, and Continue buttons, and the Table of Contents provided in the left-hand frame.

Navigation Buttons

The navigation buttons provided by the SRTE in the top frame will be visible when they are not explicitly "hidden" by an activity via the "hideLMSUI" element within the manifest. When an activity explicitly hides any of the buttons, they will not appear in the top frame.

The following section describes the behavior of the navigation buttons when they are not hidden:

  • Quit - This button will issue an Exit All navigation request to the SRTE's Sequencer; the SRTE will end a user's current attempt on a course. The user's progress and course status information for the current attempt will no longer be available if the user makes another attempt on the course.
  • Suspend - This button will issue a Suspend All navigation request to the SRTE's Sequencer; the SRTE will end a user's current attempt on a course and will also 'bookmark' the user's position within the course. The user's progress and course status information will persist. When the user starts the course again, the course will resume with the same content where they triggered the suspend.
  • Previous - This button will issue a Previous navigation request to the SRTE's Sequencer, resulting in launching the activity identified prior to the current activity in the course's activity tree.
  • Continue - This button will issue a Continue navigation request to the SRTE's Sequencer, resulting in launching the activity identified after the current activity in the course's activity tree.

Table of Contents

A table of contents (ToC) will be provided by the SRTE when a course includes Choice equal to "true" anywhere within its activity tree. When a course includes Choice equal to "false" at every level of its activity tree, the table of contents will not appear.

The following describes the items in the ToC when Choice is set to "true" in the course:

  • Selectable - Items that are selectable will be indicated with black text and the cursor will change as the user moves an input device over them.
  • Not Selectable - Items that are not selectable will be indicated by gray text or black text and a cursor that does not change when the user moves an input device over them.
Design Information


The SRTE Version 1.1.1 is composed of two distinct and separate components:

  1. SRTE Server Component: Implemented using Java Applications and Java Servlets.
  2. SRTE Client Component: Implemented using JSP, Java Applets, HTML and JavaScript.


SRTE Server Component

The SRTE Server Component is implemented using Java Servlets and Java Applications. The Servlets respond to requests from the Run-time Environment Client Component and are responsible for data model persistence as well as sequencing and parsing logic. The source files for the server component are located in the Sample_RTE\source\src directory. In addition, the JavaDoc associated with the source files is located at Sample_RTE\api\index.html. The modules that make up the Server Component are listed below:

  • ADL Validator: This module validates the imsmanifest.xml file required by SCORM Content Aggregation Content Packages. The ADL Validator also parses the XML file and returns a DOM tree representation to the SRTE Server Component. Refer to the JavaDoc Package, "org.adl.validator" for details.
  • Utilities: This module contains general purpose utility Java files. Refer to the JavaDoc Package, "org.adl.util" for details.
  • ADL Sequencer: This module handles sequencing based on the IMS Simple Sequencing Specification and SCORM 2004 4th Ed. Sequencing and Navigation Version 1.1.1. It contains classes that create an initialized activity tree, maintain the activity tree throughout the sequencing process, and handle all sequencing logic. Refer to the JavaDoc Package, "org.adl.sequencer" for details.
  • SRTE: The SRTE Package is the hub of all of the Server Component packages. It contains the logic to use the other packages when needed. It also contains Java classes that interact with Java Server Pages to implement server side functionality. Refer to the JavaDoc Packages,"org.adl.samplerte.server" and "org.adl.samplerte.util" for details.
  • Data Models: This package contains the classes that maintain the SCORM Run-Time Environment Data Model and the Navigation Data Model. Refer to the JavaDoc Package "org.adl.datamodels" for details.
SRTE Client Component (Java Files)

The SRTE Client Component consists of a sample user interface implemented in HTML and JavaScript and the SRTE API Instance implemented as a Java Applet. The Applet is downloaded to the client when the user accesses the SRTE page through a Web browser. This Applet provides the communication to the SRTE Server Component for data model element persistence. Refer to the JavaDoc Package, "org.adl.samplerte.client", for details.

SRTE Client Component (Web Files)

The SRTE Client Component is comprised of the following modules located in the Sample_RTE\source\SampleRTE\LMSClient directory:

  • Administrative: This module contains files that change passwords, delete courses, manage global objectives, create or delete users, delete courses and clear the SRTE database. The files that make up the Administrative module are located in the "admin" directory.
  • Help: This module contains information pages on how to use the SRTE. The files that make up the Help module are located in the "help" directory.
  • Import: This module is responsible for managing the course import process. These files use the SRTE Server side Java Component to handle parsing, validation and database access. The files that make up the Import module are located in the "import" directory.
  • Includes: The SRTE uses a style sheet (*.css) that handles the interface look and feel. See sampleRTE_style.css for details. This file is located in the "includes" directory.
  • Run-Time: This module contains the files used to deliver a SCORM-conformant Content Aggregation Content Package and the SCORM API. All images used in the SRTE menu are located in the "menu-images" directory of this module. The files that make up the Run-Time module are located in the "runtime" directory.
  • Special State: This module contains information pages that describe special sequencing states. Content may not be delivered as a result of a sequencing request. In that event these pages inform the user about what state the sequencer is in. Note: These pages are included to help debug sequencing in content. SCORM does not require the presence of these pages. LMSs may handle sequencing states in their own manner. The files that make up the Special State module are located in the "specialstate" directory.

Building the SRTE

Note: You are not a required to build the SRTE before using it. It will install as a fully operational unit. You only have to build the SRTE if the source code has been changed.

Complete the SRTE full installation instructions before compiling the SRTE. The SRTE build file depends on a full apache-tomcat-5.5.20 install, along with resources provided in the installation of the SRTE. When the SRTE is built, it overwrites the previously installed SRTE. To revert back to the original SRTE, execute the un-install and then rerun the downloaded self-extracting executable.

Apache Ant must be installed before building the SRTE. For information about and downloads of Ant, refer to http://ant.apache.org/.
To build the SRTE:

  1. Open a command prompt and navigate to the directory where the SRTE is installed (Default: C:\ADL\SCORM_2004_4th_Ed_Sample_RTE_V1.1.1)
  2. Navigate to the Sample_RTE\source directory
  3. At the prompt type "ant".

The XML Ant build files will compile all Java class and jar files needed by the SRTE. In addition the files will be installed to apache-tomcat. ( Note: Restart the Tomcat server for changes to take effect. This completes the compilation of the SRTE.

  2. The Internet Explorer browser must be set to check for a new page every visit to the page. To configure the browser:
    1. Open a Microsoft Internet Explorer Browser
    2. Click on "Tools" then "Internet Options"
    3. On the "General" tab, if using Internet Explorer 6 under "Temporary Internet Files" or under the "Browsing History" section if using Internet Explorer 7, click Settings
    4. Select the "Every visit to the page" radio button if it is not already selected
    5. Click OK and then OK again.

  3. Switching between different versions of the SRTE may cause an Applet Error if Applet Caching is enabled. If this occurs, follow these steps to correct the issue:
    • For Java SE 6
      1. Open the Windows "Control Panel"

      3. Click the "General" tab at the top of the dialog box
      4. Under the "Temporary Internet Files" section, click Settings
      5. Ensure that the check box labeled "Keep temporary files on my computer" is unchecked.

Implementation Decisions
Case Sensitive URLs
  • URLs that are located in the imsmanifest.xml file and that identify launch locations and other hrefs are case sensitive in the SRTE. For example, sco.html is different from SCO.HTML and from Sco.html. If the case used to specify a file name does not match the case of the actual file, an error will result while running the course.

Maximum Allowable Course Size

  • The maximum allowable size for a course imported into the SRTE is approximately 2.0 gigabytes. Courses exceeding 2.0 gigabytes may not import correctly.

Manifest Encodings

  • Any course imported into the SRTE that contains foreign characters in the manifest, must have the imsmanifest.xml file encoded in either UTF-8 or UTF-16 (Big Endian or Little Endian).

Supported Java Versions

    • Java SE 6, Update 1 or later
